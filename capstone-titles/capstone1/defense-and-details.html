Title Defense Guide: StudentHub

A Comprehensive Strategic Roadmap, System Architecture Deep-Dive, and Mock Q&A for the Capstone Title Defense

1. The Core Defense Pitch (Slide Script)

Pitch Duration: 2 to 3 minutes. Present this clearly and confidently during the solution/architecture slides.

"Good day, members of the panel. Our project, StudentHub, is a Smart Vending Machine System and Self-Sustaining Campus Edge Infrastructure with an integrated, secure campus utility wallet and intranet Platform-as-a-Service (PaaS) hosting layer.

Rather than running a monolithic server, StudentHub virtualizes its entire infrastructure using a Proxmox VE Type-1 Hypervisor hosted locally on a campus x86 mini PC. It separates the core network gateway (OpenWrt) from the application hosting environment (Ubuntu Server running Dokploy). Network segmentation is enforced using 802.1Q VLANs trunked through a TP-Link Omada managed switch to dedicated campus wireless access points (APs).

For digital payments, the system integrates Xendit as an Operator of Payment System (OPS) registered gateway to ingest digital cash-ins (GCash, Maya, QR Ph) alongside a physical coin acceptor managed by an ESP32 microcontroller. To ensure absolute compliance with Philippine banking regulations and State Universities and Colleges (SUC) auditing rules, the system operates as a strict Closed-Loop Utility. Digital top-ups are automated, but cash-outs (refunds or withdrawals) are restricted to a manual, cashier-verified dashboard workflow. This design provides campus students with a zero-cost local deployment platform for their projects while maintaining rigorous security, bandwidth fairness through SQM CAKE, and complete administrative audit trails."

2. Four Critical Defense Arguments

Argument 1: The BSP Compliance Defense (Closed-Loop Wallet)

  • The Attack: "Are you operating an illegal bank or digital wallet without a Bangko Sentral ng Pilipinas (BSP) license? How can you hold student money?"
  • The Rebuttal: "No. The system utilizes a strict Closed-Loop Utility Architecture that exempts it from licensing requirements under BSP guidelines."
  • The Logic:
  1. BSP Circular No. 1166 (Guidelines on Electronic Money): Closed-loop systems—where the monetary value stored is used exclusively within a single merchant, entity, or closed ecosystem (in this case, the ISPSC Tagudin campus grounds for campus-specific services)—are exempt from Electronic Money Issuer (EMI) licensing.
  2. Licensed Payment Rails: We do not process credit cards or digital money directly on our servers. All digital cash-in transactions go through Xendit, which holds a formal Operator of Payment System (OPS) registration with the BSP. StudentHub is simply a merchant client utilizing Xendit’s secure, licensed API rails.
  3. Utility Point System: The money in the campus wallet is not general-purpose currency; it functions as a utility token representing pre-paid compute credits or campus connectivity.

Argument 2: The COA & SUC Asset Defense (Government Auditing)

  • The Attack: "Why make cash-outs manual and documented? Why not automate it through Xendit's disbursement API to make the system completely digital?"
  • The Rebuttal: "We enforce manual, documented cash-outs to strictly comply with the Government Accounting Manual (GAM) and the Commission on Audit (COA) rules governing State Universities and Colleges (SUCs)."
  • The Logic:
  1. Public/IGP Fund Classification: ISPSC Tagudin is a state institution. Any money generated by the PisoWiFi network or collected for student orgs becomes part of the school's Income Generating Project (IGP) or official trust funds.
  2. No Automated Disbursements: Under COA rules, public funds cannot be disbursed or refunded automatically from a digital system without a verifying officer, physical signatures, and a formal disbursement voucher. Automatic digital cash-outs would constitute a serious auditing violation.
  3. Physical Paper Trail: When a student requests a cash-out, the system freezes their balance and generates a single-use voucher code. The student presents this to the campus cashier or org treasurer, who verifies the database state, pays the student cash, and files the physically signed voucher. This ensures absolute transparency and fraud prevention.

Argument 3: The System Isolation & Hypervisor Defense (Proxmox Security)

  • The Attack: "Why complicate the hardware by installing Proxmox and multiple VMs? Why not host everything on a single Ubuntu Server to reduce performance overhead?"
  • The Rebuttal: "We use Proxmox VE to guarantee strict security isolation, sandboxed software testing, and high availability for the campus network."
  • The Logic:
  1. Virtual Isolation of Routing and Applications: If the student application container (Dokploy) gets compromised or suffers a memory leak, the core routing VM (OpenWrt) remains unaffected. Students will not lose internet access or network connectivity due to application failures.
  2. Prevention of Docker Firewall Leaks: Running Docker on the same kernel as a network gateway is a well-known vulnerability. Docker automatically manipulates iptables rules, bypassing standard firewall blocks. By separating OpenWrt (routing VM) and Dokploy/Docker (application VM), OpenWrt retains total control over the firewall ruleset without Docker interference.
  3. Local Dev Playbook: Proxmox provides snapshotted restore points. If a student developer misconfigures a system configuration, the administrator can restore the container state within seconds, preserving campus uptime.

Argument 4: The Network Congestion & Resource Allocation Defense (QoS, AP, & ISP)

  • The Attack: "How will your network survive 50 to 100 concurrent students accessing the platform simultaneously on a limited ISP link?"
  • The Rebuttal: "We implement Layer 3 traffic shaping via SQM CAKE on segmented 802.1Q VLANs, backed by dedicated physical Access Points (APs) and an independent ISP backhaul."
  • The Logic:
  1. SQM CAKE (Common Applications Kept Equal): Unlike traditional FIFO queues that suffer from bufferbloat, CAKE provides flow-fairness and host-fairness. It ensures that a student downloading a large file cannot choke the connection for students loading text-based learning materials or processing transactions.
  2. Trunked VLAN Segmentation: Student client traffic is completely separated from admin operations and local vending machines at Layer 2. Administrative interfaces are inaccessible to students, preventing brute-force attacks and network scanning.
  3. Multi-AP Channel Planning: Multiple TP-Link Omada APs are deployed with non-overlapping channels (2.4GHz channels 1/6/11 and 5GHz width restrictions) to eliminate co-channel interference and handle high client density across campus common areas.

3. Technical Mock Q&A: Deep-Dive

Category A: Virtualization & Hypervisor (Proxmox VE)

"Proxmox VE is a Type-1 hypervisor. It runs directly on the bare metal, providing near-native hardware performance while allowing us to partition a single physical mini PC into multiple virtual environments.

  1. Logical and Security Segmentation: Our router (OpenWrt) runs in a dedicated Virtual Machine (VM) with hardware network interfaces passed through, while our application stack (Ubuntu/Dokploy) runs in a separate VM. If a student-deployed application crashes the Docker daemon, the network gateway remains operational.
  2. Snapshotting and Backups: It enables instant snapshots and replication. We can test major routing or ledger updates and roll them back in seconds if they fail, ensuring zero downtime for the campus network.
  3. Resource Quotas: We can hard-allocate CPU cores, RAM limits, and disk space to the application server (e.g., 4 cores, 8GB RAM) and the routing server (e.g., 2 cores, 2GB RAM), preventing resource starvation."

"For nested virtualization (running Proxmox VE as a VM inside VMware Workstation on a Windows 11 host), we configure the CPU type to 'host' (Host CPU Passthrough).

  1. Instruction Passthrough: This exposes physical CPU virtualization extensions (Intel VT-x/AMD-V) to the nested Proxmox hypervisor, allowing it to run KVM-based virtual machines.
  2. Nested Performance Overhead: Nested virtualization increases address translation overhead because Second-Level Address Translation (SLAT/EPT) must occur twice. While this reduces disk and memory I/O speed, it is highly sufficient for our staging and testing environment.
  3. Stability Controls: On Windows 11 hosts, we ensure that Virtualization-Based Security (VBS) and Core Isolation are configured properly to avoid hypervisor conflicts, which sometimes manifest as vCPU crashes in nested configurations."

"Our target production hardware is a refurbished x86 Mini PC (such as an HP EliteDesk or Dell OptiPlex Mini) with an Intel Core i5 (8th Gen, 6 Cores), 16GB of DDR4 RAM, and a 512GB NVMe SSD.

  • Hypervisor Host (Proxmox VE): Reserves 1 Core and 2GB RAM for hypervisor overhead and host monitoring.
  • Gateway Router VM (OpenWrt x86): Allocated 1 Core, 1GB RAM, and a 4GB virtual disk. This is more than sufficient for routing, DHCP, DNS, and openNDS operations.
  • Application & PaaS VM (Ubuntu Server + Dokploy): Allocated 4 Cores, 12GB RAM, and a 450GB virtual disk. This hosts the database, MQTT broker, Laravel ledger, and student application deployments."

"We implement a multi-tiered backup strategy:

  1. Automated Snapshot Backups: Proxmox is configured to run weekly snapshot backups of all VMs/LXCs to an external USB storage drive attached to the mini PC.
  2. Offsite Configuration Backup: The OpenWrt configuration file (/etc/config/) and the Laravel database SQL dumps are exported daily to a secure, off-campus cloud storage bucket (or a local admin PC).
  3. Hardware Swappability: Since the entire system is virtualized, in the event of a total motherboard failure, we can restore the backup images onto any standard x86 computer in less than an hour. We do not have custom hardware dependencies on the main host."

Category B: Platform-as-a-Service & Containers (Dokploy & Docker)

#### Q7: "How can you provision valid SSL certificates for local domains (*.studenthub.ph) that resolve on an intranet without public IP exposure?"

  • Answer: "We use the Let's Encrypt DNS-01 Challenge protocol managed automatically by Traefik inside the Dokploy VM.
  1. Why HTTP-01 Fails: Standard HTTP-01 challenges require Let's Encrypt to connect to our server on port 80 from the public internet. Since our server resides behind a local network (CGNAT / private IP), this is impossible.
  2. DNS-01 Mechanics: Traefik connects to our public DNS provider (Cloudflare) via an API token and temporarily creates a TXT record (e.g., _acme-challenge.studenthub.ph) containing a verification token.
  3. Local Resolution with Valid SSL: Let's Encrypt verifies the TXT record on the public DNS servers, confirms our ownership of the domain, and issues a wildcard SSL certificate (*.studenthub.ph). Traefik stores this certificate locally. Even though our services resolve to local IP addresses (Split-Horizon DNS), user browsers recognize the certificate as valid and secure."

"Dokploy is an open-source, self-hosted Platform-as-a-Service (PaaS) built on Docker. It acts as a local alternative to platforms like Heroku or Vercel.

  1. Deployment Dashboard: It provides student developers with a simple web interface where they can deploy applications (Node.js, Laravel, Python, databases) directly from GitHub or via zip file uploads.
  2. Automated Containerization: Dokploy automatically builds the code into a Docker image, configures Traefik for reverse proxying, and routes a local domain name (e.g., myapp.studenthub.ph) to the container.
  3. Zero External Dependencies: It runs entirely on our local Proxmox server, eliminating the need for student developers to purchase domain names, SSL certificates, or cloud hosting services."

"Dokploy leverages Docker's network isolation:

  1. Virtual Bridge Networks: Each student project is deployed on its own isolated Docker bridge network. Containers in project A cannot access containers in project B unless explicitly linked.
  2. Reverse Proxy Routing (Traefik): Student applications do not expose their internal ports (e.g., port 3000 or 8000) directly to the host network. Instead, all traffic passes through Dokploy's integrated Traefik reverse proxy running on ports 80 and 443. Traefik inspects the incoming HTTP Host header (e.g., myapp.studenthub.ph) and routes the traffic internally to the correct container, eliminating port collision issues."

"Our virtual design partitions the server logically:

  • VM 100 - Router Gateway (OpenWrt x86): Operates on the network edge. It manages WAN, DHCP leases for all subnets, Split-Horizon DNS, CAKE SQM, and the openNDS captive portal.
  • VM 101 - App Platform (Ubuntu Server 24.04 LTS): Hosts the Dokploy PaaS manager. Inside VM 101:
  • Traefik Container: The entrypoint reverse proxy handling routing and wildcard SSL.
  • Laravel Ledger Container: The core transaction engine, ledger database, named account controller, and cashier panel.
  • Node.js MQTT Bridge Container: Listens for physical coin pulse topics, validates message hashes, and triggers Laravel APIs or local OpenWrt firewall commands via ndsctl.
  • Mosquitto Broker Container: The MQTT queue managing lightweight hardware-software messaging.
  • PostgreSQL Container: A persistent database storing user records, wallet histories, and transaction states."

Category C: Layer 2/3 Networking & Wi-Fi Topology (VLANs, Switches, & APs)

"Instead of creating a separate virtual bridge interface for every single VLAN, we use a single VLAN-Aware Linux Bridge (vmbr0) in Proxmox.

  • In the /etc/network/interfaces file, the physical network port (e.g., eno1) is set to manual mode with no IP.
  • The bridge vmbr0 is created with the physical port as its member and the attribute bridge-vlan-aware yes enabled.
  • Virtual machines can be assigned to the bridge with a specific VLAN tag (e.g., Tag 30 for the Student client VLAN) directly in the Proxmox UI.
  • This setup allows Proxmox to act as a virtual managed switch, passing tagged Ethernet frames directly to the physical switch."

"The switch is configured with a strict Port/VLAN matrix to separate management, client, and IoT traffic:

  1. VLAN Allocation: We define VLAN 10 (Management), VLAN 20 (Canteen/SSIDs), VLAN 30 (PisoWiFi Clients), and VLAN 40 (Admin/Dokploy).
  2. Trunk Ports (Tagged): The port connected to the Proxmox host and the port connected to the physical Access Points are configured as Trunk Ports. They accept tagged frames for all VIDs (10, 20, 30, 40) and have a native PVID of 1 (or 10 for management).
  3. Access Ports (Untagged): Ports connected to non-VLAN-aware hardware (like the ESP32 vending controller or administrative PCs) are set as Access Ports. For example, the ESP32 port is set as an untagged member of VLAN 40 with its PVID set to 40. Any incoming untagged traffic from the ESP32 is instantly tagged with VLAN 40 by the switch."

| Switch Port | Connected Device | VLAN Mode | PVID | Tagged VIDs | Untagged VIDs |

| :--- | :--- | :--- | :--- | :--- | :--- |

| Port 1 | Proxmox Host (vmbr0) | Trunk | 10 | 10, 20, 30, 40 | 10 |

| Port 2 | TP-Link Omada AP | Trunk | 10 | 20, 30 | 10 (AP Mgmt) |

| Port 3 | ESP32 Vending Controller | Access | 40 | N/A | 40 |

| Port 4 | Cashier Dashboard PC | Access | 40 | N/A | 40 |

"We use TP-Link Omada Access Points (such as the EAP225-Outdoor or EAP610) that support Multi-SSID mapping:

  1. SSID Broadcast: The AP broadcasts two distinct SSIDs: StudentHubPisoWiFi and StudentHubAdmin.
  2. VLAN Tag Mapping:
  • The StudentHub_PisoWiFi SSID is mapped directly to VLAN 30. Any client connecting to this network has their packets tagged with VLAN 30.
  • The StudentHub_Admin SSID is mapped to VLAN 40 (secured via WPA3 Enterprise/AES).
  1. Power over Ethernet (PoE) Management: The APs are powered directly by the SG2008P switch using 802.3af/at PoE. This eliminates the need for separate power injectors and allows us to reboot the APs remotely by cycling port power through the switch dashboard."

"To ensure maximum throughput and stability under heavy loads:

  1. 2.4GHz Band Planning: We restrict broadcasts to the three non-overlapping channels: 1, 6, and 11, using a 20MHz channel width. Channel assignments are staggered geographically across APs.
  2. 5GHz Band Planning: We use 40MHz channel widths on non-DFS channels (e.g., 36, 44, 149) to maximize capacity while avoiding overlap.
  3. Transmit Power Adjustments: We tune the transmit power of each AP to minimize coverage overlap, preventing client devices from clinging to distant APs (sticky client issue) and encouraging clean roaming."

"We utilize a dedicated commercial business fiber plan (e.g., PLDT or Globe Business Fiber 100Mbps symmetrical) funded directly by the PisoWiFi network revenues.

  1. Primary WAN Setup: The fiber modem is connected directly to the designated WAN port of our OpenWrt routing VM.
  2. Bandwidth Shaping (SQM): OpenWrt is configured with SQM (Smart Queue Management) using the CAKE qdisc on the WAN interface, limited to 95% of the line rate (95Mbps) to control the bottleneck.
  3. WAN Redundancy (Future Expansion): OpenWrt's mwan3 package is pre-installed. If a secondary low-cost backup link (such as a 4G/5G gateway) is added, OpenWrt will automatically route administrative traffic through the backup link during a primary fiber outage."

Category D: Captive Portal & Session Lifecycle (openNDS & BinAuth)

"openNDS integrates with the Linux kernel packet filtering subsystem via nftables (or legacy iptables) in OpenWrt:

  1. Redirection Chains: Upon startup, openNDS inserts prerouting rules that redirect port 80 (HTTP) traffic from unauthenticated clients on VLAN 30 to the local captive portal daemon on port 2050.
  2. Walled Garden Pass-through: It allows pre-authenticated traffic to reach our local splash page (portal.studenthub.ph) and designated local services by bypassing the redirection rule.
  3. Firewall Authorization: When a client is authenticated (e.g., via the Laravel API calling ndsctl auth <MAC> <minutes>), openNDS removes the redirection rule for that specific MAC address, allowing their packets to route to the WAN interface."

"When a device connects to StudentHub_PisoWiFi and gets a DHCP lease (e.g., 10.30.0.150), the client operating system attempts to fetch a specific web page:

  • Apple iOS/macOS: http://captive.apple.com/hotspot-detect.html (looks for <title>Success</title>)
  • Android: http://connectivitycheck.gstatic.com/generate_204 (looks for an HTTP 204 response)
  • Windows: http://www.msftconnecttest.com/connecttest.txt

Since the client is unauthenticated, openNDS intercepts these HTTP probes and returns an HTTP 302 redirect pointing to http://portal.studenthub.ph:2050/. The client OS detects the redirect, identifies the network as captive, and automatically opens the Captive Network Assistant (CNA) mini-browser displaying our Laravel splash page."

"We configure the Walled Garden rules in the /etc/config/opennds configuration file:

  1. Subnet Whitelisting: We add the IP subnet of our application VM (VLAN 40) to the allowed destination ports.
  2. Required openNDS Configurations:

`text

list allowedtrustedusers '10.40.0.0/24'

list allowed_ports '80'

list allowed_ports '443'

`

  1. Pre-Auth Access: This allows unauthenticated student clients on VLAN 30 to access the Laravel backend, retrieve their wallet balance, and use free local services (such as the campus e-library) hosted on Dokploy without paying for internet access."

"The session lifecycle is managed by pairing openNDS's BinAuth script with the Laravel backend database:

  1. Deauthentication Trigger: When a student clicks 'Pause' on the splash page (or their session idle-timeouts), the Laravel API or the OpenWrt daemon triggers a client deauthentication command: ndsctl deauth <MAC>.
  2. BinAuth Script Execution: openNDS executes our configured script (binauth_log.sh) and passes session parameters:
  • $1: Event Type (clientdeauth, idledeauth)
  • $2: Client MAC address
  • $5: Session start timestamp (Unix Epoch)
  • $6: Session end timestamp (Unix Epoch)
  1. Time Calculations: The script sends a secure HTTP POST request containing these variables to the Laravel API. Laravel calculates:

$$\text{Seconds Consumed} = \text{Session End} - \text{Session Start}$$

$$\text{Unused Seconds} = \text{Total Allocated Seconds} - \text{Seconds Consumed}$$

  1. Ledger Credit Update: Laravel updates the student's wallet database record, crediting back the Unused Seconds for future usage. The user's firewall state is then reverted to the blocked/redirected status."

"We handle this using a two-pronged approach:

  1. Per-SSID Persistence: In modern iOS and Android versions, the randomized MAC address generated by a device is persistent for a given SSID. As long as our SSID name (StudentHub_PisoWiFi) does not change, the device will present the same MAC address every time it reconnects.
  2. Named Account Mapping (Fallback): For students who manually reset their network settings (which changes their randomized MAC), we require them to log in with their Student ID and PIN (Named Account). The Laravel backend associates their active MAC address with their permanent student ledger account upon login. Their credits are tied directly to their database profile, not their physical device MAC address."

Category E: Payments & Financial Auditing (Xendit, ESP32, and COA)

"Direct developer API access with GCash or Maya requires a registered business corporation, hefty security deposits, and a minimum transaction volume history. Xendit operates as an authorized Operator of Payment System (OPS) aggregator. They provide a unified API endpoint that handles GCash, Maya, and QR Ph under a single integration, allowing us to deploy local digital cash-ins legally under the Student Council's campus-authorized IGP initiative without complex corporate setups."

"The hardware vending machine operates on a secure physical-to-digital bridge:

  1. ESP32 Microcontroller: An ESP32 is wired to a multi-coin acceptor (which recognizes ₱1, ₱5, and ₱10 coins).
  2. Pulse Signal Interception: When a coin is inserted, the acceptor sends electrical pulses (e.g., 5 pulses for ₱5) to the ESP32 GPIO pins.
  3. Debounce and Counting: The ESP32 code counts the pulses, implementing a hardware debounce delay to filter out static electrical noise.
  4. MQTT Communication: The ESP32 connects to our local Wi-Fi (VLAN 40) and publishes a JSON payload containing the credit amount and a cryptographically signed HMAC token (to prevent payload spoofing) to our Mosquitto broker:

topic: campus/vending/pulse

  1. Node.js Processing: The Node.js subscriber validates the HMAC signature, confirms the physical vending machine ID, and calls the Laravel API to instantly update the user's wallet database record."

"To ensure strict compliance with COA trust fund rules, we enforce a manual checkout process:

  1. Withdrawal Request: A student requests a cash-out of ₱50 from their campus wallet web interface.
  2. Balance Freeze: The Laravel API locks the user's database row, subtracts ₱50, and creates a record marked as Pending Clearance.
  3. Voucher Generation: The system displays a unique, expiring QR code/voucher token on the student's screen.
  4. Cashier Verification: The student visits the Student Council treasurer or campus cashier. The cashier scans the QR code using their administrative dashboard.
  5. Database Validation: The cashier dashboard verifies the voucher status directly with the Laravel database. If valid, the cashier hand-delivers ₱50 cash to the student.
  6. Ledger Clearance: The cashier clicks 'Clear Voucher' on their dashboard. The system updates the transaction state to Cleared, logging the cashier's user ID, timestamp, and generating a printable paper receipt for physical filing."

Category F: Component Integration (ML & Mapping)

"The ML component is a Python-based regression model (built using scikit-learn) running as a lightweight container in our application environment.

  1. Predictive Bandwidth Scaling: It reads historical network telemetry (hourly bandwidth usage, DHCP lease durations, active client counts, and session metrics).
  2. Saturation Prediction: It predicts periods of peak campus bandwidth saturation (e.g., exam weeks or lunch hours).
  3. Automated QoS Adjustment: When saturation is predicted, it signals the OpenWrt API to dynamically lower the maximum bandwidth cap in SQM CAKE, ensuring that background downloads are throttled and priority learning applications remain fast and responsive."

"The mapping component is integrated into the Laravel Admin Dashboard using Leaflet.js:

  1. Spatial Network Layout: It renders an interactive, offline-cached map of the campus grounds.
  2. AP Topology Overlay: It plots the locations of our TP-Link Access Points and color-codes them in real-time (Green for active, Red for offline) by polling their ping status.
  3. User Density Maps: By parsing active DHCP leases and AP association lists, the map displays a visual heatmap of active user density, helping administrators identify dead zones or physical locations where additional AP hardware is needed."
